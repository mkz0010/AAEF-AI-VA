# v0.6.18 Static Fixture Validator Minimal Scaffold Design

Version: v0.6.18 candidate  
Status: static fixture validator minimal scaffold design; does not authorize runtime execution

## Purpose

This checkpoint defines a static fixture validator minimal scaffold design for
AAEF-AI-VA.

v0.6.17 defined validator scaffold planning. This checkpoint narrows that plan into
a future minimal read-only scaffold design, including module boundaries, CLI boundary,
input model, output model, planned function boundaries, failure result model, and
registration conditions before validator code is implemented.

This checkpoint is minimal scaffold design only.

This checkpoint does not implement fixture schemas, implement fixture validators,
implement negative tests, generate fixture files, create runnable configuration, create
Docker Compose files, pull images, start containers, bind ports, authorize Docker
execution, authorize runtime execution, run scanners, inject credentials, authorize
customer-target operation, create customer deliverables, provide certification,
provide legal approval, or provide audit opinion.

## Public-safe design boundary

The public design boundary is:

- AI output is treated as a request, not authority.
- AI may generate `tool_action_request`.
- Gates decide execution.
- Minimal scaffold design is not validator implementation.
- Planned modules are not implemented modules.
- Planned CLI boundary is not a working CLI.
- Planned function boundaries are not executable functions.
- Fixture schemas are not implemented by this checkpoint.
- Fixture validators are not implemented by this checkpoint.
- Negative tests are not implemented by this checkpoint.
- Fixture files are not generated by this checkpoint.
- Static fixtures are not runtime records.
- Static fixtures are not live evidence records.
- Static fixtures are not customer deliverables.
- No Docker Compose file is created by this checkpoint.
- No Docker image is pulled by this checkpoint.
- No container is started by this checkpoint.
- No port is opened by this checkpoint.
- No scanner is run by this checkpoint.
- Docker execution remains blocked.
- Runtime execution remains blocked.
- Scan execution remains blocked.
- External network testing remains blocked.
- Customer-target operation remains blocked.
- Report delivery remains separately gated.

The design must not expose private advanced feature details, protected private review
materials, market-comparison materials, or confidential commercial strategy.

## Design proposition

~~~text
A future static fixture validator should start as a minimal read-only scaffold that
loads static fixture files, validates planning-only structure and safety boundaries,
emits review-only results, fails closed on uncertainty, and never authorizes
execution, scanning, delivery, or customer-target operation.
~~~

This proposition is intentionally non-executing.

It creates a reviewable minimal design before implementation.

## Planned module boundary

A future minimal validator may be organized around these planned modules:

| Planned module | Planned responsibility |
| --- | --- |
| `static_fixture_validator` | Top-level read-only validator module |
| `fixture_loader` | Load static fixture files only |
| `manifest_checks` | Validate fixture manifest shape |
| `node_checks` | Validate required node envelopes |
| `reference_checks` | Validate reference integrity |
| `trace_checks` | Validate scenario trace order and reachability |
| `safety_checks` | Validate no-secret, no-runtime, no-runnable, no-customer boundaries |
| `failure_model` | Represent blocking review failures |
| `review_output` | Emit review-only validation result |

These are planned module boundaries, not implemented modules.

## Planned CLI boundary

A future CLI boundary may be:

~~~text
python tools/validate_static_lab_fixtures.py --fixture-dir <path>
~~~

The planned CLI must be read-only.

The planned CLI must not:

- run Docker,
- pull images,
- start containers,
- bind ports,
- run scanners,
- access external targets,
- inject credentials,
- create customer deliverables,
- authorize execution,
- authorize delivery.

The CLI boundary is not implemented by this checkpoint.

## Planned input model

Future validator inputs should be static fixture paths only.

Allowed planned inputs:

- fixture directory path,
- fixture manifest file path,
- static fixture node file paths,
- optional local schema definition path,
- optional review-output path.

Disallowed inputs:

- live scan output,
- raw customer artifacts,
- real credentials,
- real secrets,
- external network responses,
- runtime process output,
- customer deliverables,
- runnable Docker Compose files,
- executable scanner commands,
- customer target identifiers.

Input boundary violations should fail closed.

## Planned output model

Future validator outputs should be review-only validation results.

Allowed planned outputs:

- validation status,
- failure count,
- failure categories,
- affected fixture references,
- human-readable reasons,
- blocking status,
- review recommendation,
- non-proof statement reference.

Disallowed outputs:

- execution authorization,
- scanner authorization,
- Docker authorization,
- delivery authorization,
- customer-ready report,
- production-readiness claim,
- compliance claim,
- legal approval,
- audit opinion.

Validator output must not authorize execution.

| `check_no_overclaiming` | Check certification/legal/audit/production claims absence |
| `emit_review_result` | Emit review-only result |

These function boundaries are not implemented by this checkpoint.

## Failure result model

Future failure results should include:

| Field | Purpose |
| --- | --- |
| `failure_id` | Stable failure identifier |
| `failure_category` | Failure category |
| `severity` | Blocking severity |
| `fixture_ref` | Affected fixture or node |
| `field_ref` | Affected field when available |
| `message` | Human-readable reason |
| `blocking` | Whether advancement is blocked |
| `suggested_review` | Human review recommendation |
| `non_proof_statement_ref` | Non-proof linkage |

Every unsafe or uncertain failure should be blocking.

## Fail-closed behavior design

The minimal scaffold should fail closed when:

- required data is missing,
- unknown fixture node appears,
- references are broken,
- trace order is invalid,
- synthetic data statement is missing,
- execution boundary statement is missing,
- runtime marker is not false,
- runnable configuration appears,
- secret-like content appears,
- customer-like content appears,
- delivery implication appears,
- overclaiming appears,
- validator cannot safely decide.

Fail-closed behavior should prefer human review over permissive assumptions.

## Planned registration conditions

The future validator should not be registered in `tools/run_all_local_checks.py`
unless:

- minimal scaffold design is accepted,
- schema draft planning is accepted,
- negative test planning is accepted,
- implementation is read-only,
- implementation does not run Docker,
- implementation does not run scanners,
- implementation does not access external targets,
- implementation does not authorize delivery,
- failure behavior is blocking by default,
- negative tests exist before positive sample fixture generation.

Registration is not runtime authorization.

## Future implementation order

Future implementation should proceed in this order:

1. accept minimal scaffold design,
2. create read-only validator module skeleton,
3. create failure result model,
4. implement manifest loading,
5. implement required-node checks,
6. implement reference integrity checks,
7. implement trace integrity checks,
8. implement no-runtime checks,
9. implement no-runnable-configuration checks,
10. implement no-secret checks,
11. implement customer-like content checks,
12. implement overclaiming checks,
13. add negative tests,
14. register validator in local checks,
15. only then consider positive fixture generation.

Positive fixture generation should not precede negative tests.
