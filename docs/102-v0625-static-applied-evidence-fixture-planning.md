# v0.6.25 Static Applied Evidence Fixture Planning

Version: v0.6.25 candidate  
Status: static fixture planning only; does not authorize runtime execution

## Purpose

This checkpoint defines static applied evidence fixture planning for AAEF-AI-VA.

v0.6.23 defined the AAEF applied evidence package design. v0.6.24 defined the four
minimum applied evidence scenarios. v0.6.25 defines the static fixture artifacts that
should later populate those scenarios before any fixture generation, applied evidence
package generation, local-lab diagnostic execution, scanner execution, or delivery
authorization begins.

This checkpoint is fixture planning only.

This checkpoint does not generate fixture files, generate applied evidence packages,
create runnable configuration, create Docker Compose files, pull images, start
containers, bind ports, authorize Docker execution, authorize runtime execution, run
scanners, inject credentials, authorize customer-target operation, create customer
deliverables, provide certification, provide legal approval, or provide audit opinion.

## Public-safe design boundary

The public fixture planning boundary is:

- AI output is treated as a request, not authority.
- AI may generate `tool_action_request`.
- Gates decide execution.
- Fixture planning is not fixture generation.
- Fixture planning is not applied evidence package generation.
- Fixture planning is not scanner execution.
- Static fixtures are not live evidence.
- Dispatch and non-dispatch remain separated from gate decision.
- Non-execution evidence is first-class evidence.
- Static/mock fixture evidence is not proof of diagnostic accuracy.
- Real local-lab execution remains blocked by this checkpoint.
- Customer-target operation remains blocked.
- Delivery authorization remains separately gated.

Model output is not authority.

## Fixture planning proposition

~~~text
Each applied evidence scenario should have a static fixture plan that names the
required artifacts, identifies the link fields between artifacts, records the expected
gate / dispatch / result posture, and preserves non-execution evidence without
claiming runtime execution, diagnostic accuracy, audit sufficiency, legal sufficiency,
or delivery authorization.
~~~

This proposition is intentionally non-executing.

## Planned fixture root

Future static fixtures may use a public-safe path such as:

~~~text
examples/applied-evidence/static-fixtures/
  permitted-safe-diagnostic/
  denied-out-of-scope-request/
  human-approval-required/
  not-executed-expired/
~~~

Generated or private run outputs should remain outside the public repository unless a
separate public-safe decision is made.

Candidate private paths include:

~~~text
private-not-in-git/applied-evidence-runs/
private-not-in-git/applied-evidence-fixture-drafts/
private-not-in-git/applied-evidence-review-records/
~~~

This checkpoint does not create those fixture directories.

## Common fixture artifact set

Every future scenario fixture set should include:

| Planned artifact | Purpose |
| --- | --- |
| `tool_action_request.example.json` | AI-generated diagnostic intent, not authority |
| `gate_decision.example.json` | Gate evaluation and authorization boundary |
| `dispatch_decision.example.json` | Dispatch or non-dispatch decision |
| `execution_result.example.json` or `non_execution_result.example.json` | Result posture |
| `evidence_event.example.json` | Linkable evidence event |
| `review_summary.example.md` | Reviewer-facing explanation |
| `non_proof_statement.example.md` | What the fixture does not prove |

The artifact set is planned, not generated.

## Scenario identifier compatibility list

The planned static fixture set covers these exact scenario ids:

- `permitted-safe-diagnostic`
- `denied-out-of-scope-request`
- `human-approval-required`
- `not-executed-expired`

These identifiers are planning identifiers only. This checkpoint does not generate
scenario fixtures, evidence packages, live evidence, customer deliverables, or runtime
execution authorization.

## Common identifier linkage

Future fixtures should preserve stable linkage:

| Link | Expected relationship |
| --- | --- |
| `request_id` | generated by `tool_action_request` |
| `linked_request_id` | used by `gate_decision` |
| `decision_id` | generated by `gate_decision` |
| `linked_decision_id` | used by `dispatch_decision` |
| `dispatch_decision_id` | generated by `dispatch_decision` |
| `linked_dispatch_decision_id` | used by result artifact |
| `result_id` | generated by execution or non-execution result |
| `linked_result_id` | used by `evidence_event` |
| `evidence_event_id` | referenced by `review_summary` |

Identifier linkage should be structurally checkable later.

## `tool_action_request` fixture planning

A future `tool_action_request.example.json` should include:

- `request_id`,
- `scenario_id`,
- `requested_tool`,
- `action_type`,
- `target_scope`,
- `requested_parameters`,
- `purpose`,
- `rationale`,
- `risk_level`,
- `principal_context`,
- `actor_context`,
- `credential_ref` if applicable,
- `generated_by_ai`,
- `timestamp`,
- `non_authorization_statement`.

The request must remain non-authoritative.

## `gate_decision` fixture planning

A future `gate_decision.example.json` should include:

- `decision_id`,
- `scenario_id`,
- `linked_request_id`,
- `decision_result`,
- `reason`,
- `policy_version`,
- `rule_version`,
- `trusted_inputs_used`,
- `untrusted_inputs_excluded`,
- `deciding_component`,
- `timestamp`,
- `non_authorization_statement`.

The gate decision is the authorization boundary record.

## `dispatch_decision` fixture planning

A future `dispatch_decision.example.json` should include:

- `dispatch_decision_id`,
- `scenario_id`,
- `linked_decision_id`,
- `dispatch_attempted`,
- `dispatched_tool`,
- `blocked_reason`,
- `execution_boundary`,
- `gateway_component`,
- `timestamp`,
- `non_authorization_statement`.

Dispatch and non-dispatch must be explicit.

## Result artifact fixture planning

Future result artifacts should be scenario-specific.

A future `execution_result.example.json` should be used only for the
`permitted-safe-diagnostic` scenario and should include:

- `result_id`,
- `scenario_id`,
- `linked_dispatch_decision_id`,
- `execution_occurred`,
- `result_status`,
- `result_summary`,
- `sanitized_output_ref`,
- `runtime_boundary`,
- `timestamp`.

A future `non_execution_result.example.json` should be used for denied, held, expired,
or not-dispatched scenarios and should include:

- `result_id`,
- `scenario_id`,
- `linked_dispatch_decision_id`,
- `execution_occurred`,
- `non_execution_reason`,
- `blocked_component`,
- `review_required`,
- `timestamp`.

Non-execution evidence is first-class evidence.

## `evidence_event` fixture planning

A future `evidence_event.example.json` should include:

- `evidence_event_id`,
- `scenario_id`,
- `linked_request_id`,
- `linked_decision_id`,
- `linked_dispatch_decision_id`,
- `linked_result_id`,
- `event_type`,
- `event_summary`,
- `evidence_created_at`,
- `integrity_notes`,
- `non_proof_statement_ref`.

The evidence event links the chain for review.

## `review_summary` fixture planning

A future `review_summary.example.md` should explain:

- what AI requested,
- what gate evaluated,
- what was dispatched or not dispatched,
- whether execution occurred,
- what evidence event links the chain,
- how the AAEF five questions are answered,
- what the scenario does not prove.

The review summary is not a customer deliverable.

## `non_proof_statement` fixture planning

A future `non_proof_statement.example.md` should state that the fixture does not prove:

- vulnerability detection accuracy,
- production readiness,
- implementation conformance,
- audit sufficiency,
- legal sufficiency,
- compliance certification,
- external-framework equivalence,
- customer-target authorization,
- delivery authorization,
- safety guarantee.

Non-proof statements should be present for all scenarios.

## Fixture-to-AAEF five questions planning

Every future scenario fixture should answer:

| AAEF question | Fixture source |
| --- | --- |
| Who or what acted? | `tool_action_request`, `dispatch_decision`, gateway component |
| On whose behalf? | principal context and operator context in `tool_action_request` |
| With what authority? | `gate_decision`, policy version, rule version |
| Was the action allowed at the point of execution? | `dispatch_decision`, execution boundary, result artifact |
| What evidence proves what happened? | `evidence_event`, result artifact, `review_summary` |

The mapping should be present in each `review_summary.example.md`.

## Future static fixture generation readiness

Static fixture generation should not begin until:

- this fixture planning is accepted,
- reviewer walkthrough and AAEF five questions mapping are accepted,
- structural validator planning is accepted,
- naming and identifier linkage rules are accepted,
- non-proof statements are included,
- public/private artifact placement is confirmed.

Fixture generation should start as static/mock evidence, not live diagnostic evidence.

## Future structural validator planning hooks

A future structural validator should be able to check:

- required artifact presence,
- required field presence,
- scenario id consistency,
- request id linkage,
- decision id linkage,
- dispatch id linkage,
- result id linkage,
- evidence event link integrity,
- result artifact type consistency,
- contradiction checks,
- non-proof statement presence.

Validator success must not imply semantic correctness, evidence sufficiency,
assessment sufficiency, production readiness, audit sufficiency, legal sufficiency,
implementation conformance, certification, compliance, or external-framework
equivalence.
