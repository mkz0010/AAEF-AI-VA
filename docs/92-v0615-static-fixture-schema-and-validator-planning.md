# v0.6.15 Static Fixture Schema and Validator Planning

Version: v0.6.15 candidate  
Status: static fixture schema and validator planning; does not authorize runtime execution

## Purpose

This checkpoint defines static fixture schema and validator planning for AAEF-AI-VA.

v0.6.14 defined static lab scenario fixture planning. This checkpoint defines how
future fixture artifacts should be shaped, linked, and validated before any sample
fixture artifacts are generated or committed.

This checkpoint is schema and validator planning only.

This checkpoint does not generate fixture files, create runnable configuration,
create Docker Compose files, pull images, start containers, bind ports, authorize
Docker execution, authorize runtime execution, run scanners, inject credentials,
authorize customer-target operation, create customer deliverables, provide
certification, provide legal approval, or provide audit opinion.

## Public-safe design boundary

The public planning boundary is:

- AI output is treated as a request, not authority.
- AI may generate `tool_action_request`.
- Gates decide execution.
- Fixture schemas are planning records, not generated fixtures.
- Validator planning is not validator implementation.
- Static fixtures are not runtime records.
- Static fixtures are not live evidence records.
- Static fixtures are not customer deliverables.
- No fixture file is generated by this checkpoint.
- No Docker Compose file is created by this checkpoint.
- No Docker image is pulled by this checkpoint.
- No container is started by this checkpoint.
- No port is opened by this checkpoint.
- No scanner is run by this checkpoint.
- Docker execution remains blocked.
- Runtime execution remains blocked.
- Scan execution remains blocked.
- External network testing remains blocked.
- Customer-target operation remains blocked.
- Report delivery remains separately gated.

The planning must not expose private advanced feature details, protected private
review materials, market-comparison materials, or confidential commercial strategy.

## Planning proposition

~~~text
Before any static fixture artifacts are generated, the project should define the
fixture schema, required nodes, reference rules, scenario trace rules, and validator
failure categories that keep fixtures synthetic, non-executing, non-secret,
non-runnable, and review-bound.
~~~

This proposition is intentionally non-executing.

It creates a validation planning layer before fixture generation.

## Fixture manifest schema planning

A future `fixture_manifest` should include:

| Field | Required posture |
| --- | --- |
| `scenario_fixture_id` | Required, stable, unique within fixture set |
| `scenario_name` | Required, human-readable |
| `scenario_version` | Required |
| `scenario_status` | Required, static planning status |
| `fixture_set_type` | Required, static lab scenario fixture |
| `fixture_nodes` | Required ordered node list |
| `synthetic_data_statement` | Required |
| `execution_boundary_statement` | Required |
| `validator_expectation` | Required |
| `reviewer_walkthrough_ref` | Required or explicitly deferred |
| `non_proof_statement_ref` | Required |
| `advancement_state` | Required, non-executing |

A fixture manifest is not an execution manifest.

## Fixture node schema planning

Future fixture nodes should use a common node envelope:

| Field | Required posture |
| --- | --- |
| `fixture_node_id` | Required |
| `fixture_node_type` | Required |
| `fixture_node_status` | Required |
| `source_checkpoint_ref` | Required |
| `synthetic_data_statement` | Required |
| `execution_boundary_statement` | Required |
| `input_refs` | Required, may be empty |
| `output_refs` | Required, may be empty |
| `review_status` | Required |
| `non_proof_statement_ref` | Required |

The node envelope should keep fixture nodes reviewable without making them executable.

## Required fixture node types

The future fixture schema should require these node types:

| Node type | Purpose |
| --- | --- |
| `candidate_profile_fixture` | References local lab candidate profile planning |
| `static_sketch_fixture` | References static Compose design sketch planning |
| `diagnostic_context_fixture` | Describes approved synthetic diagnostic context |
| `ai_request_fixture` | Shows non-authoritative `tool_action_request` |
| `gate_decision_fixture` | Shows gate decision and reason |
| `expected_evidence_fixture` | Shows expected evidence categories |
| `reviewer_question_fixture` | Shows reviewer questions |
| `non_proof_fixture` | Shows what is not demonstrated |
| `scenario_trace_fixture` | Links nodes into a reviewable trace |

The schema should not require runtime records.

- `decision_reason`,

The gate decision fixture should keep execution authorization false.

## Expected evidence fixture schema planning

The future `expected_evidence_fixture` should include expected categories only.

Allowed expected evidence categories include:

- candidate profile evidence,
- static sketch evidence,
- non-runnable syntax evidence,
- network boundary evidence,
- port binding intent evidence,
- synthetic data evidence,
- credential exclusion evidence,
- expected review evidence,
- non-proof evidence.

Expected evidence is not live evidence.

## Scenario trace validation planning

A future validator should verify that the scenario trace links:

~~~text
candidate_profile_fixture
  -> static_sketch_fixture
  -> diagnostic_context_fixture
  -> ai_request_fixture
  -> gate_decision_fixture
  -> expected_evidence_fixture
  -> reviewer_question_fixture
  -> non_proof_fixture
~~~

Scenario trace validation should confirm:

- every referenced node exists,
- every required node type is present,
- node order is valid,
- input and output references are consistent,
- the gate decision references the AI request,
- expected evidence references the gate decision,
- reviewer questions reference the scenario,
- non-proof statement is reachable from the manifest,
- no runtime node is required.

The trace remains a review aid.

## Synthetic data validation planning

A future validator should reject or block fixtures that appear to include:

- real customer names,
- real target hostnames,
- real customer IP addresses,
- real credentials,
- real tokens,
- real assessment output from customer systems,
- raw sensitive artifacts from real systems,
- customer deliverable content,
- private advanced feature details.

Synthetic data validation supports public-safe fixture generation.

## No-secret validation planning

A future validator should check for:

- credential-like fields,
- token-like fields,
- private key markers,
- password-like fields,
- API-key-like fields,
- session-like fields,
- cookie-like fields,
- environment secret fields,
- committed secret placeholders that look real.

No-secret validation should fail closed.

## No-runtime validation planning

A future validator should check that all runtime markers remain false.

Required false markers include:

~~~text
fixture_generated = false

static_sketch_runnable = false
docker_compose_file_created = false
docker_compose_executed = false
~~~

No-runtime validation should block any fixture that implies execution.

## No-runnable-configuration validation planning

A future validator should reject or block fixture content that includes:

- runnable Docker Compose syntax,
- Docker CLI commands,
- container startup commands,
- scanner commands,
- image pull instructions,
- host bind mount instructions intended for execution,
- real port binding instructions,
- external target instructions,
- credential injection instructions.

No-runnable-configuration validation prevents static fixtures from becoming execution
instructions.

## Validator failure categories

Future validator failure categories may include:

| Failure category | Meaning |
| --- | --- |
| `missing_required_node` | Required fixture node is missing |
| `invalid_reference` | Fixture reference is broken |
| `invalid_trace_order` | Scenario trace order is invalid |
| `missing_synthetic_data_statement` | Synthetic data boundary missing |
| `runtime_marker_not_false` | Runtime marker is not false |
| `runnable_configuration_detected` | Runnable configuration detected |
| `secret_like_content_detected` | Secret-like content detected |
| `customer_like_content_detected` | Customer-like content detected |
| `missing_non_proof_statement` | Non-proof statement missing |
| `execution_implication_detected` | Fixture implies execution |

Validator failures should block advancement.

## Future validator registration planning

Future validator registration should require:

- schema planning accepted,
- fixture generation still disabled unless separately approved,

- no-runtime checks included,
- no-secret checks included,

This checkpoint does not implement the future fixture validator.
