### Exact issue body candidate

#### Candidate body: Context

~~~markdown
## Context

AAEF-AI-VA is an Applied Implementation that exercises AAEF-style authority and evidence boundaries in an AI-assisted vulnerability assessment setting.

This issue proposes a narrow public-safe handback lesson for AAEF main review. It is limited to evidence/interface-level learning from the Applied Implementation boundary.

## Proposed public-safe handback lesson

AAEF-AI-VA provides a narrow Applied Implementation lesson: evidence can preserve the distinction between AI-generated requests and execution authority.

AI output is a request, not authority.

Validator output can support structural review, but validator output is not authority.

Gate decisions remain the authority-relevant boundary for execution permission.

Dispatch and non-dispatch both benefit from evidence.

## Why this is evidence/interface-level only

The proposed lesson is not a scanner implementation, production readiness claim, compliance claim, audit opinion, legal conclusion, certification claim, diagnostic completeness claim, or external-framework equivalence claim.

The lesson is about how evidence can link request, decision, execution posture, non-execution posture, and reviewer traceability.

## AAEF five-questions mapping

The lesson maps to the AAEF five questions:

1. Who or what acted?
2. On whose behalf?
3. With what authority?
4. Was the action allowed at execution time?
5. What evidence proves what happened?

AAEF-AI-VA shows that those questions can remain reviewable even when an AI system proposes a diagnostic action, because the AI request, gate decision, dispatch or non-dispatch posture, and evidence can remain distinct.

## Authority boundary

Model output is not authority.

AI-generated tool requests do not authorize execution by themselves.

Authorization-relevant decisions remain at the gate boundary.

Validator output is not authority.

Validator success does not grant execution permission.

## Non-execution evidence boundary

Blocked, denied, human-review-required, and non-dispatched outcomes remain meaningful evidence.

Non-execution evidence can show that a proposed action did not become an authorized execution.

## Static public sample boundary

Static public samples are orientation artifacts, not live customer evidence.

They are not production assurance, diagnostic completeness proof, certification, legal advice, audit opinion, compliance claim, or external-framework equivalence.

## What is intentionally excluded

This handback excludes implementation details, patent-sensitive detail, private artifacts, customer material, credential material, scanner output, commercial Tool Gateway design, deployment runbooks, detailed PoC templates, pricing or contract material, delivery authorization material, and paid/NDA adoption package content.

It also excludes any proposal to promote AAEF-AI-VA into AAEF Core, Profile, or a Practical Package.

## Non-goals

This issue does not request adoption of AAEF-AI-VA implementation details.

This issue does not request a production-readiness finding.

This issue does not request a certification, audit, legal, compliance, or equivalence conclusion.

This issue does not request changes to AAEF main text in this issue itself.

## Requested AAEF main review outcome

Please review whether this Applied Implementation evidence/interface lesson should inform future AAEF discussion around request/authority separation, gate decisions, non-execution evidence, and reviewer traceability.

A safe outcome may be to keep this as an Applied Implementation boundary reference only, or to open a later scoped AAEF main issue/PR if a maintainer decides that an evidence/interface lesson should be reflected in AAEF main.

## No-submission and no-execution boundary note

This text was prepared as an internal AAEF-AI-VA candidate. Only the human maintainer may decide whether to open an AAEF main issue.

No runtime execution, scanner execution, Docker execution, credential injection, customer-target operation, or delivery authorization is implied by this handback.

~~~
