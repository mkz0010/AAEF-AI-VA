# v0.6.21 Static Fixture Validator Required-node Check Planning

Version: v0.6.21 candidate  
Status: static fixture validator required-node check planning; does not authorize runtime execution

## Purpose

This checkpoint defines required-node check planning for the static fixture validator.

v0.6.20 added a read-only validator scaffold at `tools/validate_static_lab_fixtures.py`.
v0.6.21 defines the next planned validation boundary: checking that a static fixture
set has a manifest and the required fixture node types before deeper schema,
reference, trace, safety, negative-test, or positive fixture work begins.

This checkpoint is required-node check planning only.

This checkpoint does not implement complete fixture schemas, implement complete
fixture validators, implement required-node checks, implement negative tests, generate
fixture files, create runnable configuration, create Docker Compose files, pull
images, start containers, bind ports, authorize Docker execution, authorize runtime
execution, run scanners, inject credentials, authorize customer-target operation,
create customer deliverables, provide certification, provide legal approval, or
provide audit opinion.

- Required-node check planning is not required-node check implementation.
- The v0.6.20 validator scaffold remains read-only.
- Fixture schemas are not completed by this checkpoint.

- Negative tests are not implemented by this checkpoint.

- Static fixtures are not runtime records.

- No Docker Compose file is created by this checkpoint.
- No Docker image is pulled by this checkpoint.
- No container is started by this checkpoint.
- No port is opened by this checkpoint.
- No scanner is run by this checkpoint.
- Docker execution remains blocked.
- Runtime execution remains blocked.
- Scan execution remains blocked.

The planning must not expose private advanced feature details, protected private

## Planning proposition

~~~text
The first future validator check after the read-only scaffold should be a
fail-closed required-node check that verifies the presence of a fixture manifest and
required static fixture node types, emits review-only results, and never authorizes
execution, scanning, Docker activity, delivery, or customer-target operation.
~~~

This proposition is intentionally non-executing.

## Planned manifest expectation

A future required-node check should expect a fixture manifest.

The planned manifest expectation is:

| Manifest item | Planned check |
| --- | --- |
| `fixture_manifest` | Manifest exists |
| `scenario_fixture_id` | Manifest has stable scenario identifier |
| `scenario_status` | Manifest is static/review-only |
| `fixture_nodes` | Manifest lists fixture nodes |
| `synthetic_data_statement` | Manifest confirms synthetic-only posture |
| `execution_boundary_statement` | Manifest confirms no-execution posture |
| `non_proof_statement_ref` | Manifest links to non-proof statement |

This checkpoint does not implement manifest parsing.

## Required fixture node set

Future required-node checks should require these node types:

| Required node type | Purpose |
| --- | --- |
| `candidate_profile_fixture` | References local lab candidate profile planning |
| `static_sketch_fixture` | References static Compose design sketch planning |
| `diagnostic_context_fixture` | Describes approved synthetic diagnostic context |
| `ai_request_fixture` | Shows non-authoritative `tool_action_request` |
| `gate_decision_fixture` | Shows gate decision and reason |
| `expected_evidence_fixture` | Shows expected evidence categories |
| `reviewer_question_fixture` | Shows reviewer questions |
| `non_proof_fixture` | Shows what is not demonstrated |
| `scenario_trace_fixture` | Links nodes into a reviewable trace |

No runtime node type should be required.

## Missing-node failure planning

A future validator should produce blocking failures for missing required nodes.

Planned missing-node failure categories include:

- `missing_fixture_manifest`,
- `missing_candidate_profile_fixture`,
- `missing_static_sketch_fixture`,
- `missing_diagnostic_context_fixture`,
- `missing_ai_request_fixture`,
- `missing_gate_decision_fixture`,
- `missing_expected_evidence_fixture`,
- `missing_reviewer_question_fixture`,
- `missing_non_proof_fixture`,
- `missing_scenario_trace_fixture`.

Missing-node failures should fail closed.

## Review-only result planning

Future required-node check results should remain review-only.

Allowed planned result fields:

- `validator_name`,
- `validator_version`,
- `validation_status`,
- `fixture_dir`,
- `manifest_status`,
- `required_node_status`,
- `missing_required_nodes`,
- `failure_categories`,
- `blocking_failures`,
- `runtime_boundary`,
- `non_authorization_statement`.

Disallowed result meanings:

- execution authorization,
- scanner authorization,
- Docker authorization,
- delivery authorization,
- customer-ready report,
- production-readiness claim,
- compliance claim,
- legal approval,
- audit opinion.

Required-node check output must not authorize execution.

## Fail-closed behavior planning

A future required-node check should fail closed when:

- fixture directory is missing,
- fixture path is not a directory,
- fixture manifest is missing,
- manifest cannot be parsed safely,
- fixture node list is missing,
- required node type is missing,
- required node type is ambiguous,
- validator cannot safely decide.

Fail-closed behavior should prefer human review over permissive assumptions.

## Read-only implementation boundary

Future required-node check implementation must remain read-only.

It must not:

- create fixture files,
- modify fixture files,
- generate positive fixture artifacts,
- generate customer deliverables,
- run Docker,
- pull images,
- start containers,
- bind ports,
- run scanners,
- access external targets,
- inject credentials,
- authorize execution,
- authorize delivery.

Any future write behavior requires separate review.

## Future implementation sequence

Future implementation should proceed in this order:

1. accept required-node check planning,
2. add missing manifest failure category if not already present,
3. add required node type constants,
4. add read-only manifest discovery,
5. add read-only manifest parsing with fail-closed behavior,
6. add required node presence checks,
7. add missing-node blocking failures,
8. add negative tests for missing manifest,
9. add negative tests for each missing required node,
10. run all local checks,
11. only then consider reference integrity check planning.

Positive fixture generation should not precede missing-node negative tests.
