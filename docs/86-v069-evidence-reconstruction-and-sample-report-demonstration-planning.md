# v0.6.9 Evidence Reconstruction and Sample Report Demonstration Planning

Version: v0.6.9 candidate  
Status: evidence reconstruction and sample report demonstration planning; does not authorize runtime execution

## Purpose

This checkpoint defines public-safe planning for evidence reconstruction and sample
report demonstration in AAEF-AI-VA.

v0.6.8 defined a non-executing demo scenario and reviewer walkthrough. This
checkpoint defines how that walkthrough can later show evidence linkage and report
packet boundaries without enabling execution.

This checkpoint is planning only.

This checkpoint does not authorize runtime execution, scanning, credential injection,
customer-target operation, production deployment, certification, legal approval, or
audit opinion.

## Public-safe design boundary

The demonstration must stay within public-safe AAEF-AI-VA principles:

- AI output is treated as a request, not authority.
- AI may generate `tool_action_request`.
- Gates decide execution.
- Assessment actions must be scoped, authorized, and evidenced.
- AI-visible context must be approved, sanitized, and bounded.
- Runtime execution remains blocked.
- Scan execution remains blocked.
- Customer-target operation remains blocked.
- Report delivery requires separate review and authorization.

The demonstration must not expose private advanced feature details, protected
private review materials, market-comparison materials, or confidential commercial
strategy.

## Demonstration proposition

The demonstration proposition is:

~~~text
A reviewer can trace a non-executing assessment story from approved diagnostic
context to AI request, gate decision, evidence expectation, finding review, report
finding demonstration, report packet review, and delivery authorization boundary.
~~~

This proposition is intentionally non-executing.

It explains reviewability and traceability without implying a working scanner,
customer-target operation, production readiness, or commercial PoC readiness.

## Evidence reconstruction scope

Evidence reconstruction planning should show:

1. what approved diagnostic context was available,
2. what AI generated as `tool_action_request`,
3. which gate evaluated the request,
4. why the gate kept execution blocked or documentation-only,
5. what evidence would be expected if a later authorized path existed,
6. how a finding candidate would remain review-bound,
7. how a report finding demonstration would remain non-deliverable,
8. how a report packet demonstration would require review,
9. how delivery authorization remains separate,
10. what the demonstration does not prove.

The scope is demonstration planning only.

## Evidence reconstruction chain

A sample evidence reconstruction chain may include these non-executing nodes:

| Node | Demonstration role |
| --- | --- |
| `scope_summary` | Explains scope and exclusions |
| `diagnostic_context_summary` | Explains approved AI-visible context |
| `normalization_loss_summary` | Explains removed, summarized, or uncertain signals |
| `tool_action_request_record` | Shows AI request generation without authority |
| `gate_decision_summary` | Shows gate decision and reason |
| `expected_evidence_summary` | Shows evidence that would be expected |
| `finding_candidate_summary` | Shows candidate finding remains review-bound |
| `finding_review_summary` | Shows review status and remaining questions |
| `report_finding_demo` | Shows report finding demonstration content |
| `report_packet_demo` | Shows report packet demonstration boundary |
| `delivery_authorization_boundary` | Shows delivery is not automatic |
| `non_proof_statement` | Shows what is not demonstrated |

These nodes are planning concepts at this stage.

They are not runtime records.

## Evidence-to-report transition

The evidence-to-report transition should remain review-bound.

Recommended transition states:

| State | Meaning |
| --- | --- |
| `candidate_recorded` | Finding candidate exists but is not report-ready |
| `needs_more_information` | Review requires more context |
| `reviewed_for_demo` | Demonstration review completed |
| `promoted_to_report_finding_demo` | Report finding demonstration created |
| `report_packet_demo_recorded` | Report packet demonstration created |
| `delivery_authorization_required` | Delivery requires separate authorization |
| `not_deliverable` | Demonstration packet must not be treated as deliverable |

The transition must not imply automatic promotion or delivery.

## Reviewer walkthrough for evidence and report

The reviewer should be able to ask:

- What approved context did AI use?
- What request did AI generate?
- Which gate evaluated the request?
- Why did execution remain blocked?
- What evidence would be expected?
- What evidence is missing?
- What uncertainty remains?
- How did the finding candidate enter review?
- What review status was assigned?
- Why is the report finding only a demonstration?
- Why is the report packet not deliverable?
- What additional authorization would be required for delivery?

These questions support review without claiming assessment completion.

## Evidence sufficiency posture

The demonstration should show evidence sufficiency posture, not evidence sufficiency
as an audit conclusion.

Allowed posture labels:

| Label | Meaning |
| --- | --- |
| `illustrative_only` | Demonstration content only |
| `insufficient_for_assessment_conclusion` | Not enough for assessment conclusion |
| `requires_human_review` | Review required |
| `requires_more_observation` | More observation needed |
| `blocked_from_execution` | Execution remains blocked |
| `not_deliverable` | Not authorized for delivery |
| `demo_packet_only` | Demonstration packet only |

The demonstration must not claim audit sufficiency or compliance sufficiency.

## Human review and delivery boundary

Human review and delivery authorization remain separate gates.

The demonstration should show:

- finding review gate,
- report review gate,
- report packet review gate,
- delivery authorization gate,
- delivery boundary,
- reason delivery remains separate from report generation.

A report-like output should not become a deliverable merely because it exists.
