## Candidate text draft

The following text is an internal candidate only. It is not final handback text, not submittable text, not AAEF main issue text, not AAEF main pull request text, not release-note text, and not document-change text.

~~~text
AAEF-AI-VA may provide a narrow Applied Implementation lesson for AAEF main at the evidence/interface level.

The public-safe lesson is that evidence can preserve the distinction between an AI-generated request and the authority to execute an action. In this pattern, AI output is a request, not authority. A gate or authorization decision remains the authority-relevant boundary for execution permission. Evidence can link the AI-generated request, the gate decision, execution or non-execution posture, and reviewer traceability.

This lesson is useful for the AAEF five questions: who or what acted, on whose behalf, with what authority, whether the action was allowed at execution time, and what evidence supports the account of what happened. Dispatch and non-dispatch both benefit from evidence. Blocked, denied, human-review-required, and non-dispatched outcomes are meaningful evidence, not absence of evidence.

Validator output can support structural review of public examples, but validator output is not execution authority. Static public samples are orientation artifacts, not live customer evidence, scanner output, production assurance, diagnostic completeness proof, certification, legal advice, audit opinion, compliance claim, or external-framework equivalence.

AAEF-AI-VA should remain an Applied Implementation boundary example. Any handback to AAEF main should avoid implementation details, patent-sensitive detail, private artifacts, customer material, credential material, scanner output, commercial Tool Gateway design, deployment runbooks, detailed PoC templates, pricing or contract material, report delivery material, and paid or NDA adoption package content.
~~~
