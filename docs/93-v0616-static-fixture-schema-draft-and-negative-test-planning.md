# v0.6.16 Static Fixture Schema Draft and Negative Test Planning

Version: v0.6.16 candidate  
Status: static fixture schema draft and negative test planning; does not authorize runtime execution

## Purpose

This checkpoint defines static fixture schema draft and negative test planning for
AAEF-AI-VA.

v0.6.15 defined static fixture schema and validator planning. This checkpoint narrows
that planning into a draft field model and negative test expectation set before any
fixture schema is implemented, any fixture validator is implemented, or any fixture
artifacts are generated.

This checkpoint is schema draft and negative test planning only.

This checkpoint does not implement fixture schemas, implement fixture validators,
generate fixture files, create runnable configuration, create Docker Compose files,
pull images, start containers, bind ports, authorize Docker execution, authorize
runtime execution, run scanners, inject credentials, authorize customer-target
operation, create customer deliverables, provide certification, provide legal
approval, or provide audit opinion.

## Public-safe design boundary

The public planning boundary is:

- AI output is treated as a request, not authority.
- AI may generate `tool_action_request`.
- Gates decide execution.
- Schema draft fields are planning records, not implemented schemas.
- Negative tests are planning records, not implemented validators.
- Fixture schemas are not implemented by this checkpoint.
- Fixture validators are not implemented by this checkpoint.
- Fixture files are not generated by this checkpoint.
- Static fixtures are not runtime records.
- Static fixtures are not live evidence records.
- Static fixtures are not customer deliverables.
- No Docker Compose file is created by this checkpoint.
- No Docker image is pulled by this checkpoint.
- No container is started by this checkpoint.
- No port is opened by this checkpoint.
- No scanner is run by this checkpoint.
- Docker execution remains blocked.
- Runtime execution remains blocked.
- Scan execution remains blocked.
- External network testing remains blocked.
- Customer-target operation remains blocked.
- Report delivery remains separately gated.

The planning must not expose private advanced feature details, protected private
review materials, market-comparison materials, or confidential commercial strategy.

## Planning proposition

~~~text
Before implementing fixture schemas or validators, the project should define draft
fields and negative test expectations that make unsafe, secret-like, customer-like,
runtime-implying, runnable, or overclaiming fixture content fail review.
~~~

This proposition is intentionally non-executing.

It creates a reviewable draft layer before validator implementation.

## Scenario trace draft rules

A future scenario trace draft should link:

~~~text
candidate_profile_fixture
  -> static_sketch_fixture
  -> diagnostic_context_fixture
  -> ai_request_fixture
  -> gate_decision_fixture
  -> expected_evidence_fixture
  -> reviewer_question_fixture
  -> non_proof_fixture
~~~

Scenario trace draft rules should require:

- every referenced node exists,
- every required node type is present,
- node order is valid,
- input and output references are consistent,
- the gate decision references the AI request,
- expected evidence references the gate decision,
- reviewer questions reference the scenario,
- non-proof statement is reachable from the manifest,
- no runtime node is required,
- no delivery node is required.

The trace draft remains a review aid.

## Missing-node negative test planning

Future negative tests should fail when required nodes are absent.

Required-node failures should include:

- missing candidate profile fixture,
- missing static sketch fixture,
- missing diagnostic context fixture,
- missing AI request fixture,
- missing gate decision fixture,
- missing expected evidence fixture,
- missing reviewer question fixture,
- missing non-proof fixture,
- missing scenario trace fixture.

Missing-node tests should fail closed.

## Broken-reference negative test planning

Future negative tests should fail when references are broken.

Broken-reference failures should include:

- manifest references missing fixture nodes,
- node input references missing sources,
- node output references missing destinations,
- AI request references missing target,
- gate decision references missing AI request,
- expected evidence references missing gate decision,
- scenario trace references missing nodes,
- non-proof statement reference missing.

Broken-reference tests should fail closed.

## Runtime-implication negative test planning

Future negative tests should fail when a fixture implies runtime behavior.

Runtime-implication failures should include:

- execution authorization set true,
- runtime marker set true,
- scan execution set true,
- live evidence indicated,
- customer target indicated,
- external network target indicated,
- delivery authorization indicated,
- generated fixture treated as customer deliverable.

Runtime-implication tests should fail closed.

## Runnable-configuration negative test planning

Future negative tests should fail when a fixture contains runnable configuration or
execution instructions.

Runnable-configuration failures should include:

- runnable Docker Compose syntax,
- Docker CLI commands,
- image pull instructions,
- container startup commands,
- scanner commands,
- port binding instructions intended for execution,
- host bind mount instructions intended for execution,
- credential injection instructions,
- external target instructions.

Runnable-configuration tests should fail closed.

## Secret-like content negative test planning

Future negative tests should fail when fixture content appears to include secrets.

Secret-like failures should include:

- real-looking password values,
- token-like values,
- private key markers,
- API-key-like values,
- session-like values,
- cookie-like values,
- environment secret values,
- production credential labels.

Secret-like tests should fail closed.

## Customer-like content negative test planning

Future negative tests should fail when fixture content appears to include customer or
real target material.

Customer-like failures should include:

- customer names,
- real target hostnames,
- real customer IP addresses,
- real assessment output from customer systems,
- raw sensitive artifacts from real systems,
- customer deliverable wording,
- production environment labels.

Customer-like tests should fail closed.

## Delivery and overclaiming negative test planning

Future negative tests should fail when fixture content implies delivery, compliance,
legal approval, audit opinion, production readiness, managed service readiness,
commercial PoC readiness, safety guarantee, or external framework equivalence.

Delivery and overclaiming tests should fail closed.
