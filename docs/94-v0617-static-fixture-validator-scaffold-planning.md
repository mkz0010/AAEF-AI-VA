# v0.6.17 Static Fixture Validator Scaffold Planning

Version: v0.6.17 candidate  
Status: static fixture validator scaffold planning; does not authorize runtime execution

## Purpose

This checkpoint defines static fixture validator scaffold planning for AAEF-AI-VA.

v0.6.15 defined static fixture schema and validator planning. v0.6.16 defined schema draft and negative test planning. This checkpoint defines the scaffold responsibilities and implementation order for a future static fixture validator before any validator code is implemented or fixture artifacts are generated.

This checkpoint is validator scaffold planning only.

This checkpoint does not implement fixture schemas, implement fixture validators, implement negative tests, generate fixture files, create runnable configuration, create Docker Compose files, pull images, start containers, bind ports, authorize Docker execution, authorize runtime execution, run scanners, inject credentials, authorize customer-target operation, create customer deliverables, provide certification, provide legal approval, or provide audit opinion.

## Scaffold responsibility model

A future static fixture validator scaffold should plan checks for fixture manifest presence, required fixture node presence, fixture node envelope shape, reference integrity, scenario trace integrity, synthetic data statements, execution boundary statements, no-secret posture, no-runtime posture, no-runnable-configuration posture, customer-like content exclusion, delivery implication exclusion, overclaiming exclusion, non-proof statement linkage, and blocking failure reporting.

The scaffold should not run tools, start containers, or validate live assessment evidence.

## Validator input boundary

Future validator inputs should be static fixture artifacts only.

Allowed planned inputs:

- `fixture_manifest`
- `candidate_profile_fixture`
- `static_sketch_fixture`
- `diagnostic_context_fixture`
- `ai_request_fixture`
- `gate_decision_fixture`
- `expected_evidence_fixture`
- `reviewer_question_fixture`
- `non_proof_fixture`
- `scenario_trace_fixture`

Disallowed inputs include live scan output, raw customer artifacts, real credentials, real secrets, external network responses, runtime process output, customer deliverables, runnable Docker Compose files, and executable scanner commands.

Input boundary violations should fail closed.

## Validator output boundary

Future validator outputs should be review records, not execution permissions.

Allowed planned outputs include validation status, failure category, affected fixture node, human-readable reason, blocking status, suggested review action, non-proof statement reference, and advancement recommendation.

Disallowed outputs include execution authorization, scanner authorization, Docker authorization, delivery authorization, customer-ready report, production-readiness claim, compliance claim, legal approval, and audit opinion.

Validator output must not authorize execution.

The stages are planning concepts, not implemented functions.

## Fail-closed behavior planning

Future validator behavior should be fail-closed.

Fail-closed behavior means missing required data, invalid references, unknown node types, runtime implication, secret-like content, customer-like content, runnable configuration, delivery implication, overclaiming, or validator uncertainty blocks advancement.

Fail-closed behavior should prefer human review over permissive assumptions.

## Failure category model

Future validator failure categories should include:

- `missing_manifest`
- `missing_required_node`
- `invalid_node_envelope`
- `invalid_reference`
- `invalid_trace_order`
- `missing_synthetic_data_statement`
- `missing_execution_boundary_statement`
- `secret_like_content_detected`
- `customer_like_content_detected`
- `runtime_marker_not_false`
- `runnable_configuration_detected`
- `delivery_implication_detected`
- `overclaiming_detected`
- `validator_uncertainty`

Every unsafe fixture failure should be blocking.

## Negative-test handling planning

Future negative-test handling should require negative cases defined before positive fixture generation, each negative class maps to a failure category, each negative case expects blocking behavior, no negative case requires runtime execution, no negative case includes real secrets, no negative case includes real customer material, no negative case includes executable scanner instructions, and negative examples remain synthetic and non-executing.

Negative-test handling is not implemented by this checkpoint.

## Validator review record planning

Future validator review records should include:

- `validator_review_id`
- `fixture_manifest_ref`
- `validation_status`
- `failure_categories`
- `blocking_failures`
- `review_recommendation`
- `advancement_state`
- `non_proof_statement_ref`

Validator review records should not become execution authorization records.

## Future implementation order

Future implementation should proceed in this order:

1. accept scaffold planning,
2. define minimal validator module boundary,
3. implement read-only fixture loading,
4. implement required-node checks,
5. implement reference checks,
6. implement no-runtime checks,
7. implement no-runnable-configuration checks,
8. implement no-secret checks,
9. implement customer-like content checks,
10. implement overclaiming checks,
11. implement negative tests,
12. register validator in local checks,
13. only then consider positive fixture generation.

Positive fixture generation should not precede negative test expectations.

## Registration gate planning

A future validator should not be registered unless scaffold planning is accepted, schema draft planning is accepted, negative test planning is accepted, validator implementation remains read-only, validator does not run Docker, validator does not run scanners, validator does not access external targets, validator does not authorize delivery, failure behavior is blocking by default, and `tools/run_all_local_checks.py` registration is reviewed.

Registration is not runtime authorization.
